> 标书检查Agent完整功能设计 — 自动生成修改建议 + 人机协同复核
> 版本：V1.0
> 编制日期：2026年3月

---

## 一、功能概述

### 1.1 自动生成修改建议

根据AI检查发现的问题，自动生成针对性的修改建议。根据问题类型自适应选择建议粒度（条款级/段落级/文档级）。

**用户场景：**
- AI初筛后发现问题，需要给出具体修改方向
- 减少人工撰写修改建议的工作量
- 加速标书修改迭代

### 1.2 人机协同复核

通过串行复核流程：AI初筛 → 人工二查 → 打回修改 → 循环直到通过，确保标书质量。

**用户场景：**
- 多轮修改后的质量把关
- 复杂问题需要人工判断
- 合规性要求必须有人工复核环节

## 四、功能详细设计

### 4.1 自动生成修改建议

#### 4.1.1 粒度自适应策略

根据问题类型自动选择建议粒度：

| 问题类型 | 粒度 | 示例 |
|----------|------|------|
| 条款缺失 | 条款级 | "建议补充：售后服务条款" |
| 星号条款响应不完整 | 条款级 | "★条款要求2年质保，建议修改为..." |
| 格式/排版错误 | 段落级 | "第二段建议调整为以下格式..." |
| 章节结构问题 | 文档级 | "建议将技术方案章节移至商务部分之前" |
| 表述歧义 | 句子级 | "此处表述建议明确为..." |

#### 4.1.2 建议生成流程

```
AI发现问题
    ↓
分析问题类型
    ↓
确定粒度
    ↓
LLM生成修改建议
    ↓
组装建议对象
    ↓
进入建议池（待用户确认）
```

#### 4.1.3 预览后应用流程

```
用户查看建议列表
    ↓
预览单个建议的原文 vs 建议内容
    ↓
选择操作：
├─ 确认应用 → 一键应用到标书
├─ 拒绝建议 → 标记为已拒绝
└─ 修改建议 → 编辑后再确认
    ↓
应用后 → 生成新版本 → 重新进入检查流程
```

### 4.2 人机协同复核（串行流程）

#### 4.2.1 复核流程定义

```
                    ┌──────────────────────┐
                    │       [草稿]         │
                    │  用户创建/修改标书   │
                    └──────────┬───────────┘
                               │ 提交审核
                               ▼
                    ┌──────────────────────┐
                    │     [AI待审]         │
                    │  AI完成初步检查       │
                    └──────────┬───────────┘
                               │ AI检查通过
                               ▼
                    ┌──────────────────────┐
                    │    [人工待审]         │
                    │  人工二查复核         │
                    └──────────┬───────────┘
                               │
                    ┌──────────┴───────────┐
                    │                       │
                  通过                      有问题
                    │                       │
                    ▼                       ▼
        ┌──────────────────┐    ┌──────────────────┐
        │    [终审]        │    │   [打回修改]     │
        │  终审人员把关     │    │  返回用户修改    │
        └──────────┬───────┘    └──────────┬───────┘
                   │                       │
                 通过                      修改后
                   │                       │
                   ▼                       ▼
        ┌──────────────────┐    ┌──────────────────┐
        │   [已完成]       │    │     [草稿]       │
        │  复核流程结束    │    │  重新提交审核    │
        └──────────────────┘    └──────────────────┘
```

#### 4.2.2 复核角色

| 角色 | 职责 | 操作权限 |
|------|------|----------|
| 投标撰写人 | 根据建议修改标书 | 创建、编辑、提交审核 |
| 初审人员 | AI初筛结果的复核 | 通过、打回、补充问题 |
| 终审人员 | 最终质量把关 | 终审通过、拒绝 |

#### 4.2.3 复核操作

**初审人员操作：**
- 查看AI发现的所有问题
- 对每个问题标记：通过/打回/需修改
- 添加人工发现的额外问题
- 填写复核意见

**终审人员操作：**
- 查看初审结果和全部问题状态
- 进行最终质量把关
- 确认通过 → 流程结束
- 拒绝 → 需说明理由

## 六、错误处理

| 场景 | 处理策略 |
|------|----------|
| LLM生成建议失败 | 降级为模板填充，提示用户"建议生成失败，请手动填写" |
| 建议应用后文档损坏 | 版本回滚，保留修改历史，用户可撤销 |
| 复核超时未处理 | 自动提醒复核人，超时3天升级通知 |
| 循环修改超过5次 | 提示"建议优化标书结构"，建议重新审视整体方案 |
