## 一、功能概述

### 1.1 历史中标案例检索

根据用户上传的招标文件，检索系统中历史中标案例，找出结构相似的中标标书作为参考，并生成差异化分析报告。

**用户场景：**
- 投标前了解同类项目的中标方案
- 参考历史案例优化当前标书
- 分析竞争对手（中标方）的优势

### 1.2 多版本标书对比

对比投标文件与招标要求的变化，或当前标书与模板的差异，帮助用户快速识别变更点。

**用户场景：**
- 确认投标文件是否响应了招标要求的变更
- 检查当前标书与公司标准模板的偏差
- 追踪标书修改历史

## 四、功能详细设计

### 4.1 历史中标案例检索

#### 4.1.1 检索入口

支持三种检索入口：

| 入口类型 | 实现方式 | 描述 |
|----------|----------|------|
| 按招标文件检索 | 语义向量检索 | 上传招标文件，系统自动匹配相似历史案例 |
| 按行业/时间检索 | 过滤+排序 | 选择行业类型和时间范围，返回该范围内的中标案例 |
| 按投标方检索 | 关键词检索 | 输入公司名称，返回该公司历史中标记录 |

#### 4.1.2 检索流程

```
用户上传招标文件 / 选择过滤条件
    ↓
语义检索 → 历史标书库
    ↓
按相似度排序，取Top N案例
    ↓
选择N个最相似案例
    ↓
LLM生成差异化分析报告
    ↓
输出：差异点列表 + 差异原因分析
```

#### 4.1.3 差异化分析报告

```markdown
## 差异化分析报告

### 本项目 vs [CASE-2025-001] XX系统建设项目

#### 相似度得分
整体相似度：85%

#### 主要差异点

| 维度 | 本项目 | 历史案例 | 差异分析 |
|------|--------|----------|----------|
| 工期 | 30天 | 28天 | 本项目工期更宽松 |
| 质保期 | 1年 | 2年 | 竞争对手提供更长质保期 |

#### 竞争对手（中标方）优势
1. 提供了2年质保期，超出招标要求
2. 工期承诺28天，展现技术实力

#### 投标建议
1. 建议将质保期提升至2年以提升竞争力
2. 工期可保持30天，无需压缩
```

### 4.2 多版本标书对比

#### 4.2.1 对比模式

| 模式 | 描述 | 数据来源 |
|------|------|----------|
| 投标vs招标 | 对比投标文件与招标要求的变化 | 投标最新版本 vs 招标文件 |
| 当前vs模板 | 对比当前标书与公司标准模板的差异 | 投标最新版本 vs 模板库 |

#### 4.2.2 对比流程

```
用户选择对比模式
    ↓

    ↓

    ↓
用户切换视图
    ↓
输出变更报告
```

## 五、核心算法

### 5.1 语义检索算法

**方案：向量语义检索 (RAG)**

1. 将招标文件解析为条款列表
2. 每条条款生成向量嵌入
3. 在历史标书库中检索相似条款
4. 聚合相似度得分，返回Top N案例

**相似度计算：**
```python
similarity = cosine_similarity(query_embedding, case_embedding)
final_score = weighted_sum([
    clause_similarity * 0.4,
    industry_match * 0.3,
    time_relevance * 0.3
])
```

### 5.2 Diff算法

**方案：段落级Diff + 变更聚类**

1. 将文档按段落分割
2. 计算相邻版本间的文本相似度
3. 低于阈值判定为变更
4. 变更类型判定：
   - 原文无、新文有 → 新增
   - 原文有、新文无 → 删除
   - 原文有新文有但相似度低 → 修改

## 七、错误处理

| 场景 | 处理策略 |
|------|----------|
| 历史案例库为空 | 提示用户"暂无历史数据，请先导入" |
| 检索无结果 | 返回空列表，提示"未找到相似案例" |
| 文档解析失败 | 返回解析错误，提示用户检查文件格式 |
| 版本数据缺失 | 标记为"数据不完整"，仅展示可用部分 |
